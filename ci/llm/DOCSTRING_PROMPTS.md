# Prompts for the "add LLM docstrings" workflow

These drive an LLM (e.g. Claude with code execution) through adding docstrings to
a Python module using `llm_docstrings.py`. Upload both files at the start.

The division of labor is the whole point:

- **The tool** finds undocumented functions, inserts docstrings at the correct
  indentation, and proves that only docstrings changed.
- **The model** reads each function and writes an accurate docstring. This part
  is not automatable — a plausible-but-wrong docstring in a codebase is worse
  than none, so the model must actually read the code.

---

## 1. Session setup prompt (paste once, at the start)

> I want to add docstrings to the functions in a Python module. I've uploaded
> `llm_docstrings.py` (a helper tool) and the target file(s).
>
> Rules for the docstrings you write:
> - Use `:param:`, `:type:`, `:return:`, `:rtype:` fields.
> - Write the body in Markdown (backticks for identifiers, shapes, math).
> - Do **not** invent behavior. Read each function's actual code before
>   documenting it. If a function is a stub, has dead code after a `return`, or
>   is unclear, say so in the docstring rather than guessing.
> - Describe what actually happens. Read the code itself and NEVER use vague statements like
>   "generated by the implementation" or "consistent with the implementation" or
>   "handling the alternate cases encoded by its conditional branches"
> - Indicate actual return types or leave them blank, don't call everything `object`
> - Only add docstrings to functions that lack one; never modify existing
>   docstrings or any code.
> - The tool prepends the header `**LLM Docstring**` automatically, so the JSON
>   you produce should contain only the body (no header, no triple quotes).
>
> Workflow, per file:
> 1. `python llm_docstrings.py scan <file>` to see what's missing.
> 2. `python llm_docstrings.py dump <file> --batch N --batch-size 20` to read
>    the source of each undocumented function (batch through all of them).
> 3. Write a `docs.json` mapping each qualified name to its docstring body.
> 4. `python llm_docstrings.py insert <file> docs.json`
> 5. `python llm_docstrings.py verify <file> --docs docs.json --show`
>    — this must report `missing=0`, `non_llm_header=0`, and
>    `code_identical=True`. If not, fix and re-run (`restore` reverts).
>
> Before editing, confirm the file's source of truth (see prompt 2). Work in
> quality-controlled batches and don't rush; accuracy matters more than speed.
> Start by scanning the file and telling me how many functions need docstrings
> and how you'll batch the work.

---

## 2. Source-of-truth check (use if the file isn't the authoritative copy)

Skip this if you're editing a local file directly. Use it when the file lives
somewhere else (a package, a repo) and you want to be sure the copy you edit
matches.

> Before writing anything, verify the copy of `<file>` you're about to edit is
> the same as my authoritative version. Pick a few distinctive fragments
> (function names, a unique line of code, an `__all__` entry) and confirm they
> appear in both. If they don't match, stop and tell me — don't edit a file that
> differs from mine.

---

## 3. Per-batch prompt (repeat until every batch is done)

> Now do batch `<N>`: run
> `python llm_docstrings.py dump <file> --batch <N> --batch-size 20`, read every
> function in the output, and add its docstring body to `docs.json`. For each
> one, work out from the code what the inputs are, what it returns, and what it
> does — including array shapes and any non-obvious behavior. Flag anything you
> had to infer rather than read directly.

---

## 4. Insert + verify prompt

> Insert the docstrings and verify:
>
> ```
> python llm_docstrings.py insert <file> docs.json
> python llm_docstrings.py verify <file> --docs docs.json --show
> ```
>
> Confirm the verify output shows `missing=0`, `non_llm_header=0`, and
> `code_identical=True`. If `code_identical` is `False`, something other than a
> docstring changed — run `python llm_docstrings.py restore <file>` and redo the
> insert. Then give me the finished file.

---

## 5. Multi-module prompt (optional)

> There are several modules to do. Handle them one at a time using the loop
> above. For very large files (hundreds of functions), do one module per
> response so each stays high quality; give me the verified file at the end of
> each before moving on. Tell me the running total as you go.

---

## Tool cheat-sheet

```
python llm_docstrings.py scan    <file>                 # list undocumented funcs
python llm_docstrings.py scan    <file> --batch-size 20 # + how many batches
python llm_docstrings.py dump    <file> --batch 0 --batch-size 20
python llm_docstrings.py keys    <file>                 # JSON skeleton to fill in
python llm_docstrings.py insert  <file> docs.json       # writes <file>.docbak backup
python llm_docstrings.py verify  <file> --docs docs.json --show
python llm_docstrings.py restore <file>                 # undo (from .docbak)
```

`insert` options: `--header "..."` to change the header, `--no-header` to omit
it, `--allow-partial` to insert only some of the missing functions (useful when
filling `docs.json` incrementally across batches — verify with `--docs` once
it's complete).

### docs.json format

```json
{
  "some_function": "One-line summary.\n\nMore detail in `Markdown`.\n\n:param x: the thing\n:type x: np.ndarray\n:return: the result\n:rtype: np.ndarray",
  "SomeClass.some_method": ":param n: ...\n:type n: int\n:return: ...\n:rtype: ..."
}
```

Keys are qualified names exactly as `scan`/`keys` print them (e.g.
`Class.method`). A property and its setter share one qualified name; a single
entry documents both. Values are the body only — the tool adds the header and
the triple quotes and fixes indentation.
